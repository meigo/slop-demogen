# demogen Design Spec

**Date:** 2026-05-12
**Status:** Brainstorming → ready for writing-plans
**Scope:** A standalone npm package, intended to live in a new repo `slop-demogen/`, that generalises `slop-plant-doctor/scripts/record-demo.ts` into a reusable library.

## Summary

A small npm package (`demogen`) that records repo-committed demo GIFs / MP4s of web apps by owning the *plumbing* of Playwright-based recording (dev-server lifecycle, viewport setup, mock-route shorthand, video capture, ffmpeg encoding with tuned defaults). The per-app *flow logic* stays as a bring-your-own Playwright callback the developer (or Claude) writes per project.

**Core decisions:**

- **Plumbing only — no LLM inside the library.** AI (Claude) is the *user* of the library; the library is pure code.
- **Bring-your-own-steps API.** Imperative Playwright callback, not a YAML/JSON DSL. Imperative wins because Claude writes the callback anyway and a DSL would be a leaky abstraction.
- **`ffmpeg` shells out for encoding** — battle-tested flags shipped as defaults. This is the genuinely scarce piece.
- **TypeScript, ESM-only.** Peer-deps on `playwright`. Calls `ffmpeg` via `execSync`; assumes it's on `PATH`.
- **`demo.gif` + `demo.mp4` at repo root by default.** Both ship; both are cheap.
- **Framework-aware dev-server presets** — Vite, Next, SvelteKit, Astro, plus an escape hatch for arbitrary commands.
- **`npx demogen` CLI** that picks up `demo.config.ts` from CWD, runs the flow, writes the outputs.

## Context

The reference implementation is `slop-plant-doctor/scripts/record-demo.ts` (135 lines). It works, but ~100 of those lines are plumbing that any future slop app would re-implement: server boot + ready probe, viewport setup, route mocking, `chromium.launch` boilerplate, ffmpeg incantation, cleanup. The flow-specific part is ~20 lines.

The slop portfolio is on track to have 5+ pattern instances (Plant Doctor shipped, Appliance Troubleshooter mid-build, Mechanical-Part Identifier and others queued). Each needs a `demo.gif` for its README. Extracting plumbing-as-library now saves re-writing it 5+ times and standardises GIF quality across the portfolio.

Comparable in the terminal-demo space: Charm's [`vhs`](https://github.com/charmbracelet/vhs) — declarative `.tape` files driving a terminal, ~14k stars, no monetisation, brand-builder for Charm. The web equivalent doesn't exist in canonical npm form. Existing options:

| Tool | Why it doesn't fill the slot |
|---|---|
| Playwright `recordVideo` | Raw WebM only; no GIF pipeline; no mocking shorthand; no dev-server lifecycle. |
| `puppeteer-recorder` | Puppeteer-specific, unmaintained since 2021. |
| `playwright-video` | Deprecated. |
| Demosmith / Supademo / Trupeer | Hosted SaaS for non-engineers. Wrong user. $40+/mo. Off-repo. |

## Tech Stack

| Concern | Choice | Notes |
|---|---|---|
| Language | **TypeScript** (ESM-only) | Matches all other slop repos. |
| Runtime | **Node ≥ 20** | For native `fetch`, ESM, top-level await. |
| Browser automation | **`playwright`** (peer-dep) | Library re-exports `chromium`, `devices` for steps callback to use. Consumer installs `playwright` themselves. |
| Video encoding | **`ffmpeg`** via `execSync` | Assumed on PATH. Library prints a friendly error if missing. |
| Build / publish | **`tsup`** → ESM + d.ts | Simple, fast, no config drama. |
| Testing | **`vitest`** | Unit tests for config validation + ffmpeg filter construction. Integration test boots a tiny static site and records. |
| Lint / format | **`prettier`** + minimal `eslint` | Match slop-plant-doctor's setup. |

## Public API

### `recordDemo(config)`

The single function the library exists to provide.

```ts
import type { Page, BrowserContext } from 'playwright';

export interface DemoConfig {
  // What to start, how to wait for it, what env to pass
  dev: DevServerConfig;

  // Where the recording navigates first
  url?: string; // default: 'http://localhost:<dev.port>'

  // Viewport + device emulation
  viewport: ViewportPreset | ViewportSpec;

  // Output paths (relative to CWD)
  output: {
    gif?: string | false;  // default: 'demo.gif'
    mp4?: string | false;  // default: 'demo.mp4'
  };

  // Trim from start / end of recording (seconds)
  trim?: { start?: number; end?: number };  // default: { start: 3 }

  // Route mocks — shorthand over page.route()
  mocks?: Record<string, MockResponse>;

  // The per-app flow. The only required user-supplied code.
  steps: (page: Page, ctx: StepContext) => Promise<void>;

  // Encoding overrides (rarely needed)
  encode?: Partial<EncodeOptions>;
}

export interface StepContext {
  fixture(dirRelativeToCwd: string): string;  // returns path to first image in dir
  base: string;                                 // dev-server base URL
  context: BrowserContext;                      // for advanced use
}

export type ViewportPreset = 'iphone-13' | 'iphone-15-pro' | 'pixel-7' | 'desktop' | 'square-1080';

export interface ViewportSpec {
  width: number;
  height: number;
  deviceScaleFactor?: number;
  isMobile?: boolean;
  userAgent?: string;
}

export interface MockResponse {
  status?: number;          // default: 200
  body: unknown;            // JSON-serialisable, or string for non-JSON
  contentType?: string;     // default: 'application/json' if body is object
  delay?: number;           // ms — emulate latency
}

export interface EncodeOptions {
  fps: number;               // default: 10
  gifWidth: number;          // default: 320
  gifMaxColors: number;      // default: 96
  gifDither: 'bayer' | 'floyd_steinberg' | 'none';  // default: 'bayer'
  mp4Crf: number;            // default: 28
}
```

### `defineConfig(config)`

Identity helper for `demo.config.ts` type inference:

```ts
import { defineConfig } from 'demogen';

export default defineConfig({
  dev: { framework: 'vite', port: 5183, env: { OPENROUTER_API_KEY: 'mock' } },
  viewport: 'iphone-13',
  mocks: {
    '/api/diagnose': { delay: 1500, body: { id: 'demo-id' } }
  },
  steps: async (page, ctx) => {
    await page.locator('input[type="file"]').nth(1).setInputFiles(ctx.fixture('tests/fixtures/plant-photos'));
    await page.waitForSelector('img[alt="Selected plant"]');
    // …
  }
});
```

### Viewport presets

Wrap Playwright's `devices` map. Each preset bakes in `viewport`, `userAgent`, `deviceScaleFactor`, `isMobile`, and the video recording size (which must equal viewport size to avoid letterboxing — a non-obvious detail).

| Preset | Size | Source |
|---|---|---|
| `iphone-13` | 390 × 720 (cropped from 844 default) | Playwright `devices['iPhone 13']` |
| `iphone-15-pro` | 393 × 720 | Playwright `devices['iPhone 15 Pro']` |
| `pixel-7` | 412 × 720 | Playwright `devices['Pixel 7']` |
| `desktop` | 1280 × 800 | Custom |
| `square-1080` | 1080 × 1080 | Custom (for Twitter / LinkedIn cards) |

Default heights are clamped to 720 — README GIFs longer than that get unwieldy.

## Dev-Server Adapters

```ts
export type DevServerConfig =
  | { framework: 'vite' | 'next' | 'sveltekit' | 'astro'; port: number; env?: Record<string, string>; cwd?: string }
  | { command: string; port: number; readyLog: string | RegExp; env?: Record<string, string>; cwd?: string }
  | { url: string };  // skip launch — server already running
```

Built-in framework presets resolve to `command` + `readyLog`:

| Framework | Resolved command | Resolved ready signal |
|---|---|---|
| `vite` | `npx vite dev --port <port>` | log contains `Local:` |
| `next` | `npx next dev --port <port>` | log contains `Ready in` |
| `sveltekit` | `npx vite dev --port <port>` | log contains `Local:` (same as Vite under the hood) |
| `astro` | `npx astro dev --port <port>` | log contains `Local` |

Implementation owns: `spawn` with `stdio: ['ignore', 'pipe', 'pipe']`, ready-log race against a 30s timeout, `SIGTERM` on cleanup, pre-warm of `/` via `fetch` (avoids first-frame-blank in lazy-route frameworks).

## Mock-Route Shorthand

```ts
mocks: {
  '/api/diagnose': { delay: 1500, body: { id: 'demo' } },
  '/api/upload': { status: 201, body: { ok: true } },
  '/static/heavy.json': { body: '{}', contentType: 'application/json' }
}
```

Compiles to a single `page.route('**/*', ...)` handler that matches the request URL against the keys (glob-aware via `minimatch` or manual `*` expansion). Anything unmatched falls through (`route.fallback()`).

Glob support:
- `/api/diagnose` matches `**/api/diagnose` (leading `**/` implicit).
- `/api/*` matches one-segment wildcards.
- `**/foo` matches anywhere.

## The ffmpeg Pipeline

This is the spec's load-bearing section — the one piece that's genuinely arcane and that the library is encoding once so consumers don't have to.

### Filter chain (default, GIF)

```
fps=<fps>,
scale=<width>:-1:flags=lanczos,
split[s0][s1];
[s0]palettegen=max_colors=<colors>[p];
[s1][p]paletteuse=dither=<dither>
```

Defaults: `fps=10, width=320, colors=96, dither=bayer`.

Rationale per flag:

| Flag | Why this value |
|---|---|
| `fps=10` | Demo GIFs don't need 30 fps. 10 fps cuts file size by 3× with no perceived loss for UI clicks/scrolls. |
| `scale=320:-1:flags=lanczos` | 320px wide reads fine in a README at full width but stays under ~2 MB. Lanczos preserves sharp text on downscale. |
| `palettegen=max_colors=96` | 96 colors (out of 256 max) is the sweet spot for UI screencasts — backgrounds + accents render cleanly without dither artifacts on solid panels. 256 colors bloats the palette ~2×; 64 colors visibly bands gradients. |
| `paletteuse=dither=bayer` | Bayer is the right dither for clean UI screencasts (low-noise, structured). `floyd_steinberg` looks better for photos but adds visible noise on flat panels. `none` produces visible banding. |

### Trim

`ffmpeg -ss <trim.start> -i <webm> ... -t <duration-trim.start-trim.end>`

Default `trim.start=3` because Playwright recordings start blank for ~1–3 s before the page is visible (browser cold-start). Trim end defaults to 0.

### MP4 path

```
ffmpeg -y -ss <start> -i <webm> -c:v libx264 -preset slow -crf 28 -movflags +faststart -pix_fmt yuv420p <mp4>
```

Rationale:
- `libx264` — universally supported, no royalty drama for OSS.
- `-crf 28` — good quality/size tradeoff for UI screencasts (lower CRF for photos/film).
- `-preset slow` — encode time doesn't matter; quality does.
- `-movflags +faststart` — playable while still loading (important for landing-page embeds).
- `-pix_fmt yuv420p` — required for Safari / older Android.

### Detection

On first run, `which ffmpeg` (POSIX) / `where ffmpeg` (Windows). If missing, print:

```
✗ demogen: ffmpeg not found on PATH
  Install: https://ffmpeg.org/download.html (macOS: `brew install ffmpeg`)
```

## CLI

```
npx demogen                    # reads ./demo.config.ts
npx demogen --config foo.ts    # custom path
npx demogen --debug            # keeps temp WebM + prints ffmpeg invocation
```

CLI is ~50 LOC: load config (via `tsx` for TS support), call `recordDemo`, print progress lines.

## Self-Hosting (Dogfood)

`slop-demogen/demo.config.ts` records a tiny demo of demogen recording itself, producing the README's `demo.gif`. Meta but useful as a smoke test and as living documentation. The "tiny demo app" is `examples/counter/` — a 30-line Vite app with a button counter.

## Open Questions

- **Mock matching library:** roll our own glob expansion, or pull `minimatch`? `minimatch` is widely used and ~10 KB; rolling our own is ~30 LOC but one less dep. Lean towards `minimatch` for correctness on edge cases.
- **Should the CLI also accept inline flow files (`npx demogen --steps flow.ts`)?** Adds flexibility but increases API surface. Skip for v1; revisit if it comes up.
- **Static site mode:** if there's no dev server (pre-built dist), should the library spin up a static file server itself, or require the consumer to do that? Probably require — keeps scope tight. `{ url: 'http://localhost:8080' }` with the consumer responsible for the server covers it.
- **Windows support:** `ffmpeg` is available on Windows but the `execSync` quoting is finicky. Add to v1.1 if requested; default targets macOS / Linux for v1.

## Out of Scope (Explicit)

- **No LLM dependencies.** Repeat for emphasis. No API key required to use this library. Ever.
- **No hosted service.** No SaaS, no account, no telemetry.
- **No competition with Demosmith.** The audience here is "developer who wants a fresh demo.gif committed to their repo." Marketing teams are not the audience.
- **No "describe in English" mode.** The user (or Claude in their terminal) writes Playwright code. The library doesn't try to be smart about flows.
